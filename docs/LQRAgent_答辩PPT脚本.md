## 第 2 页｜赛题要求与真实痛点

**页面内容**

**赛题要求**
- 面向《Python 高级语言程序设计》
- 个性化学习 · 智能答疑 · 学习路径规划 · 学习画像

**三类真实痛点（左图右文）**
1. **传统学习平台**：内容固定，无法因材施教，学生只能被动看课。
2. **通用大模型**：没有课程知识库，容易幻觉，也无法形成学习闭环。
3. **学习数据孤岛**：路径、练习、错题、资料、画像彼此独立，无法联动。

**讲解要点**
- 先点赛题要求，说明我们严格围绕赛题展开。
- 再用三类痛点引出我们的目标：**让 AI 参与每个学习环节，但每个环节都可控、可追踪、可闭环**。
- 关键词：因材施教、闭环、可控。

**视觉建议**
- 左侧：赛题关键词云或图标。
- 右侧：三类痛点用对比卡片呈现，底部用箭头指向下一页「解决方案」。

## 第 3 页｜产品定位：从「能聊天」到「能学习」

**页面内容**

**一句话定位**
> LQRAgent = 课程知识库 + 多 Agent 编排 + 学习全链路闭环

**核心闭环「问—学—练—评—改」**
- 问：聊天中智能答疑，优先检索课程知识库
- 学：AI 规划学习路径，生成讲义与资源
- 练：按知识点推荐题目，答题即时反馈
- 评：知识图谱与学习画像记录掌握度
- 改：薄弱点自动反哺下一轮路径与出题

**讲解要点**
- 强调我们不是「会聊天的搜索引擎」，而是「可信赖的学习伙伴」。
- 用闭环图说明：学生在系统里的每一次操作都会沉淀为数据，数据又反过来优化推荐与规划。

**视觉建议**
- 中央放环形/线性闭环图，五个节点用箭头连接。
- 每个节点配学生端对应页面的小图标（聊天、路径、答题、图谱、画像）。

**建议时长**：40 秒

## 第 4 页｜系统总体架构

**页面内容**

**三层架构（自上而下）**
1. **学生端（React + Vite + TypeScript）**
   - 聊天 / 学习路径 / 学习资源 / 答题 / 上传 / 学习画像 / 知识图谱
2. **Java 后端 Orchestrator（Spring Boot）**
   - WebSocket 入口 / PlanningAgent 意图识别 / Pipeline 编排 / 20+ Agent / 业务数据持久化
3. **AI Server（Python FastAPI，基于 DeepTutor）**
   - LLM 调用 / RAG 检索 / 向量库（ChromaDB）/ 知识库管理 / 文件解析

**数据流**
用户提问 → WebSocket → PlanningAgent 拆解意图 → Pipeline 编排 Agent 链 → 调用 AI Server / 数据库 → 流式返回结果 + Artifact 卡片

**讲解要点**
- 强调分层清晰：前端只负责展示，后端负责编排，AI Server 负责模型与向量能力。
- 强调「不是单模型一把梭」：用户的一句话会经过意图识别、任务拆解、多 Agent 协作后再返回。
- 提到 WebSocket 流式，让学生感受到"AI 在一步步思考"。

**视觉建议**
- 一张清晰的三层架构图，用箭头标注请求流向。
- 右侧或下方配一张聊天界面截图，标注「WebSocket 流式 + Agent 步骤条 + Artifact 卡片」。

**建议时长**：55 秒

## 第 5 页｜多 Agent 协作机制

**页面内容**

**三层 Agent 体系**
- **调度层**：PlanningAgent（LLM Function Calling，支持 8 种意图路由）
- **学习层**：Profile / LearningPath / QA / Resource / Quiz / Difficulty / Quality 等 20+ Agent
- **督导层**：ConsultationEngine 多轮协商 + SupervisorService 质量审核

**关键机制**
1. **意图识别**：问候、帮助、学习路径、资源、问答、图表、总结、评估等 8 类意图自动路由。
2. **多轮协商**：路径生成、出题设计由 Profile、LearningPath、Difficulty 等 Agent 协商完成。
3. **质量督导**：Supervisor 对路径与题目做后置审核。
4. **防幻觉**：RAG 检索优先 → 引用来源展示 → Quality Agent 后置检查。

**讲解要点**
- 这是项目的核心创新，要讲得慢、讲得透。
- 用「Agent 步骤条」截图说明：学生能看到 AI 的思考过程，增强信任感。
- 防幻觉是评委高频关注点，要重点提 RAG + 引用 + 质检三层防护。

**视觉建议**
- 左侧：Agent 分层结构图或同心圆。
- 右侧：聊天界面中展开「Agent 调用链」的截图，标注 PlanningAgent、QA Agent、RAG 检索、Quality Agent。
- 底部：RAG 引用来源卡片截图。

**建议时长**：1 分钟

## 第 7 页｜聊天学习 · 智能问答

**页面内容**

**三大使用场景**
1. **能力介绍**：「你可以做什么」→ 返回系统能力清单
2. **课程答疑**：「什么是 Python 列表推导式」→ QA Agent + RAG 检索 → 回答附带引用
3. **主动规划**：「帮我制定 Python 学习路径」→ 进入协商 → 输出结构化路径卡片

**技术亮点**
- WebSocket 流式响应
- Agent 步骤条可展开/折叠，过程透明
- 历史会话持久化
- Artifact 卡片：路径、资源、题目以结构化卡片呈现，不是纯文本

**讲解要点**
- 这是学生最高频的入口，展示时要让学生「一句话就能触发完整学习流程」。
- 重点展示 RAG 引用来源：回答下方出现「引用自课程讲义/上传资料」，体现防幻觉。
- 展示路径 Artifact：聊天里直接出现可点击的学习路径卡片。

**视觉建议**
- 页面以 2～3 张聊天界面截图为主：
  - 图 1：问候/能力清单截图
  - 图 2：答疑截图，圈出引用来源卡片
  - 图 3：生成路径卡片截图，圈出 Artifact 卡片
- 截图要高清、关键信息用红框/箭头标注。

**建议时长**：1 分钟

## 第 10 页｜资料上传 · RAG · 学习画像

**页面内容**

**资料上传与 RAG**
- 上传：PPT / PDF → 七牛云存储 → 异步解析 → 向量化入私有知识库 `kb-private-{userId}`
- 问答：RagSearchTool 检索公共库 + 私有库 → 前端展示引用来源
- 防幻觉：先检索再生成；无资料时拒答或标注「通用说明」

**学习画像**
- 画像维度：知识掌握度 · 薄弱点 · 学习风格 · 阶段目标
- 动态更新：对话后 ProfileExtractor 增量抽取
- 导出：Markdown / PDF 学习报告

**讲解要点**
- 上传是学生个性化学习的关键：把自己的课件变成 AI 可检索的知识库。
- 画像反哺：画像不是静态报告，而是持续影响路径难度和出题策略。

**视觉建议**
- 左侧：上传页截图 + 上传队列状态截图。
- 右侧：聊天中引用「上传资料」的回答截图 + 学习画像页截图（雷达图/进度条）。
- 用箭头串联：上传资料 → RAG 引用 → 更新画像。

**建议时长**：55 秒

## 第 12 页｜创新点与技术栈

**页面内容**

**四大创新点**
1. **多 Agent Pipeline + 协商/督导**：PlanningAgent 意图识别 + ConsultationEngine 多轮协商 + Supervisor 质量审核，非单轮 Prompt。
2. **学习全链路打通**：聊天 → 路径 → 资源 → 练 → 图谱 → 画像，数据互相反哺。
3. **RAG 私有知识库 + 引用可验证**：支持学生上传资料，回答附带来源，显著降低幻觉。
4. **流式体验 + Agent 步骤透明**：WebSocket 流式 + 可折叠 Agent 步骤条 + Artifact 卡片，结果可解释、可信任。

**技术栈**
- 前端：React 18 · Vite 5 · TypeScript
- 后端：Spring Boot 3.2 · Java 21 · WebSocket · JWT · MySQL 8 · Redis
- AI 服务：Python FastAPI · DeepTutor · LlamaIndex · ChromaDB
- 模型：DeepSeek（可热配置切换）
- 存储：七牛云对象存储

**讲解要点**
- 创新点要对应前面的截图，避免空泛。
- 技术栈点到为止，突出「可热配置换模型」和「WebSocket 流式」两个比赛加分项。

**视觉建议**
- 上半页：4 个创新点卡片，每卡配小图标。
- 下半页：技术栈图标云或分层列表。
- 可选：底部放管理后台「系统配置」小图，说明模型可热配置。

**建议时长**：50 秒
