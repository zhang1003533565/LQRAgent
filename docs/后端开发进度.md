## 二、每个智能体的实现方案

### 2.1 OrchestratorAgent（协调智能体）

**赛题要求**：
- 接收请求 → 路由 → **聚合结果**
- 维护请求上下文（request_id, user_id）
- **禁止交叉调用**，星型拓扑
- 使用**消息队列**（Redis Streams）进行通信

**实现方式**：

```
用户请求 → ChatWebSocketHandler / HTTP Controller
  → RequestContext.init(userId, requestId)
  → OrchestratorAgent.process()
    → 根据 intent 创建子任务列表
    → 通过 AgentBus（Redis Streams）派发到各 Agent
    → 等待全部返回
    → 聚合结果
    → QualityGate 校验
    → 返回前端
```

| 能力 | 实现来源 | 说明 |
|------|---------|------|
| 意图识别 | **🔗 LLM API** (P5-2 已有 `LlmIntentClassifier`) | 优先 LLM，降级关键词 |
| 任务派发 | **🏗️ 本地** AgentBus → Redis Streams | M2 加 |
| 结果聚合 | **🏗️ 本地** ResultAggregator | 新建 |
| 重试 | **🏗️ 本地** | QualityGate 拦截时自动重试 |
| RequestContext | **🏗️ 本地** RequestContext.java | 新建，ThreadLocal |

**关键改动**：
- `agent/RequestContext.java` — **新建**，ThreadLocal 全链路传递
- `agent/AgentBus.java` — **大改**，从 in-process 改为 Redis Streams
- `agent/RedisStreamConfig.java` — **新建**
- `agents/OrchestratorAgent.java` — **改**，加结果聚合逻辑
- `agents/OrchestratorAgent.java` — **改**，并行派发多个子任务

**策略说明**：Orchestrator 永远不调 AI，只做路由决策 + 派发 + 聚合 + 重试。

### 2.2 LearnerProfileAgent（学生画像智能体）

**赛题要求**：
- LLM Prompt 从对话抽取 ≥6 维度（知识基础、学习目标、认知风格、易错点、学习节奏、兴趣方向）
- 存储到 PostgreSQL + Chroma（语义检索）
- 增量更新

**实现方式**：

```
对话进行中
  → ai-server Memory API（`/api/v1/memory`）保存对话摘要  ← 🔗 ai-server
  → LearnerProfileAgent.process()
    → 取最近 N 条对话 + ai-server Memory 中的 profile
    → 调 LLM（`ResourceLlmGenerator` 复用）抽取 6 维度     ← 🔗 LLM API
    → 写 PostgreSQL `learner_profile` 表                  ← 🏗️ 本地
    → 推 WS `profile_patch` 事件
```

| 能力 | 实现来源 | 说明 |
|------|---------|------|
| 对话历史存储 | **🔗 ai-server** Memory API | 已有 `getMemory()` / `updateMemory()` 定义，启用 |
| 6 维度抽取 | **🔗 LLM API** | 复用 `ResourceLlmGenerator`（加 system prompt） |
| 结构化存储 | **🏗️ 本地** PostgreSQL | 已有 `learner_profile` 表，扩展字段 |
| 语义检索 | **🏗️ 本地** Chroma HTTP API | 可选，对话向量化 |
| 增量更新 | **🏗️ 本地** Java Service | 已有 `PATCH /api/profile/summary` |

**LLM Prompt**：
```
System: 从以下对话记录中提取学生画像6个维度，以JSON格式返回。
{ "knowledge_base": "初学/基础/进阶/熟练",
  "learning_goal": "目标描述",
  "cognitive_style": "视觉型/听觉型/动手型/阅读型",
  "weakness": ["易错点1", "易错点2"],
  "learning_pace": "快/中/慢",
  "interest": "兴趣方向" }
只输出JSON。
```

**关键改动**：
- `learnerprofile/service/ProfileExtractor.java` — **新建**，LLM 抽取逻辑
- `learnerprofile/service/LearnerProfileService.java` — **改**，集成抽取+ai-server Memory
- `AiServerClient.java` — **改**，启用 `getMemory()` / `updateMemory()`

### 2.3 ResourceFacadeAgent（资源生成智能体）

**赛题要求**：
- ≥5 种资源类型：**讲解文档 / 思维导图 / 练习题目 / 拓展阅读 / 代码案例**
- 可选：多模态视频/动画（脚本+图文帧）

**实现方式**：

| 资源类型 | 实现来源 | 当前状态 | 说明 |
|---------|---------|---------|------|
| ① 讲解文档 | **🔗 LLM API** | ✅ 已有 | P5-1 完成 |
| ② 练习题目 | **🔗 LLM API** | ✅ 已有 | P5-1 完成 |
| ③ 代码案例 | **🔗 LLM API** | ✅ 已有 | P5-1 完成 |
| ④ **思维导图** | **🔗 LLM API** | ❌ 新增 | LLM 生成 Markdown 列表，前端 mermaid.js 渲染 |
| ⑤ **拓展阅读** | **🔗 LLM API** | ❌ 新增 | LLM 生成延伸阅读清单+摘要 |

**为什么不用 ai-server book engine**：
ai-server 的 Book Engine (`/api/v1/book`) 是完整的出书流程（create→propose→confirm→compile→read），需要多步交互。而我们的需求是"给定知识点名称，一次返回 Markdown 内容"，直调 LLM API 更简单。

**并行生成**（P5-6）：

```java
// 5 种资源并发生成
List<ResourceType> types = List.of("LESSON", "MIND_MAP", "QUIZ", "EXTENDED_READING", "CODE_CASE");
// 并行调 LLM API → QualityGate 校验 → 合并返回
```

**思维导图 Prompt**：
```
用 Markdown 列表格式输出思维导图结构。
- 中心主题：Python 装饰器
  - 定义
  - 语法
  - 应用场景
  ...
```

**关键改动**：
- `resourcefacade/service/ResourceFacadeService.java` — **改**，加 MIND_MAP / EXTENDED_READING 类型
- `resourcefacade/service/ResourceLlmGenerator.java` — **改**，扩展 Prompt 表

### 2.4 QualityAssessmentAgent（质量评估智能体）

**赛题要求**：
- 事实性校验：LLM 自检或 RAG 比对
- 敏感内容过滤：集成讯飞内容审核 API 或本地敏感词库
- 学术规范性检查：虚假引用、绝对化用语
- 返回 pass/fail + 原因，失败→**重试一次**

**实现方式**：

| 检查项 | 实现来源 | 说明 |
|--------|---------|------|
| 事实性校验 | **🔗 LLM API** | "这段内容是否有明显错误？" |
| 敏感内容 | **🏗️ 本地** 敏感词库 + **🔗 第三方 API** (讯飞可选) | 先本地，上线后加讯飞 |
| 学术规范 | **🏗️ 本地** 规则匹配 | "参考文献[1]"+ 绝对化用语检测 |
| 重试机制 | **🏗️ 本地** Orchestrator 控制 | QualityGate 拦截时重试 |

**LLM 自检 Prompt**：
```
检查以下教学内容是否有事实错误。如果发现明显错误，输出 "FAIL: 原因"；
如果没有明显错误，输出 "PASS"。只输出一行。
```

**敏感词库**：本地 `resources/sensitive-words.txt`（200+ 条），匹配到即拦截。

**关键改动**：
- `qualityassessment/service/SensitiveFilter.java` — **新建**
- `qualityassessment/service/AcademicChecker.java` — **新建**
- `qualityassessment/service/QualityAssessmentService.java` — **改**
- `resources/sensitive-words.txt` — **新建**

### M2 — Redis Streams（消息队列）

**赛题要求**：使用 Redis Streams 或 RabbitMQ

**实现方式**：

```
Stream: agent:tasks
Message: {requestId, userId, targetAgent, taskType, payload}

Consumers: 每个 Agent 一个独立消费组
  orchestrator-worker
  learning-path-worker
  resource-facade-worker
  ...
```

| 组件 | 角色 |
|------|------|
| Redis Streams | 任务队列 |
| AgentBus.dispatch() | **写入** Streams |
| Agent 消费者 | **读取** Streams，处理，写结果到 Reply Stream |
| Orchestrator | 等全部结果 → 聚合 |

**状态**：❌ 新建。保留当前 in-process AgentBus 做开发测试，Redis Streams 用于正式部署。

**策略**：**先 Redis，后独立部署**。Redis Streams 已解耦通信层，Agent 独立 jar 只是打包方式变化。
