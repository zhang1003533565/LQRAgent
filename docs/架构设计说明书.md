# LQRAgent 架构设计说明书

> 用途：提交给架构师评审
> 项目定位：赛题 A3 软件杯 — 多智能体协同的个性化学习系统

---

## 一、系统概述

### 1.1 业务目标

一个自适应学习平台，通过**多智能体协同**实现：

1. **学习画像构建** — 对话中自动抽取学生 6 维画像（知识基础、目标、认知风格、薄弱点、学习节奏、兴趣）
2. **资源生成** — 针对知识点自动生成 5 种教学资源（讲义/代码/题目/思维导图/拓展阅读）
3. **路径规划** — 基于知识图谱 BFS 生成最短学习路径
4. **智能答疑** — 流式文字回答 + Mermaid 流程图辅助
5. **效果评估** — 行为追踪 → 薄弱点分析 → 路径动态调整

### 1.2 技术栈

| 层 | 技术 | 版本 | 用途 |
|---|------|:----:|------|
| 后端框架 | Spring Boot | 3.2.5 | REST API + WebSocket + JPA |
| 数据库 | MySQL 8 + Redis | — | 主存储 + 消息队列（预留） |
| LLM 引擎 | DeepSeek API | — | 所有文本生成 + 推理决策 |
| 前端 | React 19 + Vite 5 | — | 学生端 + 管理端 SPA |
| 构建 | Maven | — | 后端构建 |

### 2.2 与"传统 Agent 框架"的区别

```
传统 Agent 框架（LangChain / CrewAI）:
  Agent.process()
    → 写死的 switch/case
    → 调方法
    → 返回

本项目实现（LLM + Tool Calling）:
  AgentEngine.run()
    → LLM 接收:[systemPrompt, userInput, tools]
    → LLM 自己决策: "用户要生成资源，我调 generate_lesson"
    → ToolRegistry 执行工具
    → 结果喂回 LLM
    → LLM 继续决策: "还需要其他资源吗？"
    → ...循环直到 LLM 认为任务完成
```

### 2.3 核心抽象

```
Agent (interface)
  ├─ agentId()              → 唯一标识
  ├─ getSystemPrompt(task)  → LLM 角色定义
  ├─ getTools()             → 工具列表 (ToolSchema)
  ├─ registerTools(registry)→ 工具执行逻辑注册
  └─ process(task)          → 向后兼容的旧入口

AgentEngine
  └─ run(agent, task)       → LLM 推理循环

ToolRegistry
  └─ register(agentId, toolName, executor)
  └─ execute(agentId, toolName, args)

ToolSchema
  └─ toOpenAISpec()         → 转 OpenAI function calling 格式
```

## 四、推理循环的两种模式

### 4.1 模式 A：AgentEngine 全推理循环（真智能体）

```
用户: "我想学装饰器，帮我生成学习资料"
  ↓
AgentEngine.run(orchestratorAgent)
  ↓
round 1:
  LLM(系统提示词 + 用户消息 + [route_intent, greet, handle_learning_path, ...])
  → LLM 决定: 调 route_intent("我想学装饰器")
  → ToolRegistry 执行: route_intent → "resource_generate"
  → 结果入上下文
  ↓
round 2:
  LLM(系统提示词 + 原消息 + route_intent 结果 + 工具列表)
  → LLM 决定: 调 handle_resource_generate("我想学装饰器")
  → ToolRegistry 执行: → 调 ResourceFacadeService.generate()
  → LLM 生成讲义/题目/代码...
  → 结果入上下文
  ↓
round 3:
  LLM 认为任务完成
  → 返回最终答案
```

### 4.2 模式 B：降级（传统流水线）

```
未配置 LLM API Key 时:
  AgentEngine.run()
  → 检测到 apiKey 为空
  → 直接调 agent.process(task)
  → 走旧的 switch/case 逻辑
  → 返回模板结果
```

## 五、数据流

### 5.1 资源生成请求

```
前端 POST /api/resources/generate {kpId, resourceType}
  ↓
ResourceFacadeService.generate()
  ↓
callGenerateOrFallback(type, title, desc, template)
  ↓
先试 LLM:
  ResourceLlmGenerator.generate()
    → 检查 apiKey → 有则调 DeepSeek API
    → 解析返回内容
    → 成功 → 返回 LLM 内容
    → 失败 → 返回 null
  ↓
LLM 成功 → 保存 LLM 内容到 DB
LLM 失败 → 返回本地模板
```

### 5.2 答题提交请求

```
前端 POST /api/quiz/submit {kpId, answer, resourceId}
  ↓
QuizService.submit()
  ↓
1. 评分（精确匹配 100 分 / 自由文本 60 分 / 错误 0 分）
2. 更新 LearnerProfile（答题记录累加）
3. EffectAssessmentService.evaluateQuizResult()
   → 低分则插入复习节点到学习路径
4. 记录行为到 study_behavior 表
```

### 5.3 学习路径生成

```
前端 GET /api/learning-path?goal=xxx
  ↓
LearningPathService.generatePath(userId, goal)
  ↓
1. 解析 goal 找到目标知识点（如 kp_decorator）
2. 从 31 节点有向图中 BFS 找最短路径
3. 计算每步状态（completed?）
4. 保存到 learning_path + learning_path_step 表
5. 返回有序节点列表
```

## 九、技术评审要点

### 9.1 优点

1. **架构清晰** — Agent 接口抽象 + LLM 推理循环 + 工具注册中心，三层职责分离
2. **渐进迁移** — 新旧两套并存，AgentEngine 检测到无 API Key 自动降级为 process()
3. **松耦合** — Agent 不直接调别的 Agent，通过 ToolRegistry 共享工具
4. **可扩展** — 新增 Agent 只需实现 4 个接口方法 + 注册工具
5. **LLM 无关** — 兼容任何 OpenAI API 格式的 LLM（DeepSeek / GPT / Ollama 等）

### 9.2 待改进

1. **AgentBus 接线** — 已切到 AgentEngine，但 Agent 的 process() 仍是 switch/case，LLM 未接管
2. **LearningPath LLM 排序** — BFS 完没有调 LLM 做个性化排序
3. **QaAgent WS 通道** — answer 工具走 WebSocket 流式，AgentEngine 的 HTTP 调用无法直接对接
4. **前端延迟** — 前端大部分页面还在用 mock 数据
5. **测试覆盖** — 缺少单元测试和集成测试
6. **状态持久化** — AgentEngine 的上下文是内存态，宕机丢失

### 9.3 风险提示

- **LLM 延迟** — DeepSeek API 调用约 2-5s，多轮推理循环可能累积到 10s+
- **Token 消耗** — 每轮推理循环都发送完整消息历史，长对话可能 token 超标
- **降级策略** — LLM 不可用时降级到模板，但模板内容不够个性化
- **并发限制** — 当前所有请求共享一个 AgentEngine 实例，无隔离
